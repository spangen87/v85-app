## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Supabase (via server component) | Server component fetches from Supabase using authenticated session cookie; no client-side DB access |
| Next.js server → Supabase | Uses `createClient()` which inherits the user's session; RLS enforces read-only access to `games` table for authenticated users |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01-01 | Information Disclosure | `evaluation/page.tsx` — `games` table query | accept | The `games` table is read-only for authenticated users via RLS. No PII in games rows (only date, track, game_type). All authenticated users can see all games — this is intentional (no per-user isolation needed for evaluation view). |
| T-02-01-02 | Spoofing | Supabase session cookie | accept | Handled by Supabase Auth middleware and `createClient()` from `lib/supabase/server.ts`. Session cookies are httpOnly and signed. No additional mitigation needed at this layer. |
| T-02-01-03 | Denial of Service | `games` table query without limit | mitigate | Add `.limit(100)` to the `games` query: `supabase.from("games").select(...).order("date", { ascending: false }).limit(100)`. Prevents unbounded result sets from a large historic games table. |
